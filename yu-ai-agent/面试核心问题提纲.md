## 一、项目概述（必答）

### Q1：请用 2 分钟介绍一下这个项目

**答案要点**（3 句话）：
1. **定位**：基于 Spring Boot 3.4 和 Spring AI 的智能代理系统，实现自主规划能力的 AI Agent
2. **核心功能**：ReAct 模式智能体（Think-Act 循环）、RAG 知识库问答、工具调用、多轮对话记忆
3. **技术亮点**：继承链设计（模板方法模式）、手动管理工具调用上下文、文件持久化会话记忆

**STAR 模板**：
- **S**：AI Agent 项目，需要实现自主规划、工具调用的智能体
- **T**：实现 ReAct 模式，支持 RAG、工具调用、会话记忆
- **A**：采用继承链设计，手动管理消息上下文，使用 Kryo 序列化持久化
- **R**：成功实现可自主规划、调用工具、自动终止的智能体系统

### Q2：项目的核心业务流程是什么？

**答案要点**（4 步）：
1. **状态管理**：IDLE → RUNNING → FINISHED/ERROR（状态机模式）
2. **执行循环**：for (step < maxSteps && state != FINISHED)
3. **Think-Act**：think() 调用 AI 决策工具 → act() 执行工具调用
4. **终止条件**：调用 doTerminate 工具 或 达到 maxSteps

**流程图**：
```
用户输入 → 状态校验 → 执行循环 → Think(决策) → Act(执行) → 更新上下文 → 检查终止 → 返回结果
```

## 二、核心架构（重点）

### Q3：智能体的继承链设计是如何实现的？

**答案要点**（4 层）：
1. **BaseAgent**（抽象基类）：状态管理、执行循环、资源清理
2. **ReActAgent**（抽象类）：实现 step() = think() + act()
3. **ToolCallAgent**（具体类）：实现 think() 和 act()，管理工具调用
4. **YuManus**（最终类）：配置提示词和参数，可直接使用

**设计模式**：模板方法模式（定义算法骨架，子类实现具体步骤）

**关键代码**：
```java
// BaseAgent 定义骨架
public String run() {
    for (int i = 0; i < maxSteps && state != FINISHED; i++) {
        String stepResult = step();  // 抽象方法
    }
}

// ReActAgent 实现 step
public String step() {
    boolean shouldAct = think();  // 抽象方法
    if (shouldAct) return act();  // 抽象方法
    return "无需行动";
}
```

### Q4：为什么选择手动管理工具调用，而不是使用 Spring AI 的自动机制？

**答案要点**（3 点）：
1. **需求**：需要实现 ReAct 模式，手动控制 think 和 act 的分离
2. **问题**：Spring AI 自动工具调用会直接执行，无法分离决策和执行
3. **解决**：`DashScopeChatOptions.builder().withProxyToolCalls(true)` 禁用自动代理，使用 `ToolCallingManager` 手动执行

**关键代码**：
```java
// 禁用自动代理
this.chatOptions = DashScopeChatOptions.builder()
        .withProxyToolCalls(true)  // 关键：不让 Spring 代理
        .build();

// 手动执行工具
ToolExecutionResult result = toolCallingManager.executeToolCalls(prompt, toolCallChatResponse);
setMessageList(result.conversationHistory());  // 手动更新消息上下文
```

### Q7：抽象类 BaseAgent 和接口 ChatMemory 的区别？

**答案要点**（4 点）：
1. **抽象类**：有构造方法、可以有实现方法、单继承
2. **接口**：无构造方法、Java 8+ 可以有默认方法、多实现
3. **使用场景**：BaseAgent 需要共享状态和方法实现，ChatMemory 只定义契约
4. **设计原则**：抽象类用于"是什么"，接口用于"能做什么"

---

### Q8：为什么使用 ArrayList 而不是 LinkedList 存储消息列表？

**答案要点**（3 点）：
1. **访问模式**：消息列表需要顺序遍历和随机访问，ArrayList 的 get(i) 是 O(1)
2. **内存效率**：ArrayList 连续内存，缓存友好，LinkedList 节点分散
3. **实际场景**：消息列表主要是追加和遍历，很少中间插入删除

## 四、设计模式（必问）

### Q9：项目中使用了哪些设计模式？请举例说明

**答案要点**（4 个）：
1. **模板方法模式**：BaseAgent.run() 定义执行流程，step() 由子类实现
2. **策略模式**：不同工具（WebSearchTool、FileOperationTool）实现 ToolCallback 接口
3. **责任链模式**：Advisor 拦截器链（MyLoggerAdvisor、MessageChatMemoryAdvisor）
4. **工厂模式**：ToolRegistration 创建工具数组，LoveAppRagCustomAdvisorFactory 创建 Advisor

**模板方法示例**：
```java
// 模板方法：定义算法骨架
public String run(String userPrompt) {
    // 1. 校验
    // 2. 执行循环
    for (int i = 0; i < maxSteps; i++) {
        step();  // 抽象方法，由子类实现
    }
    // 3. 清理
}
```

---

### Q10：为什么使用模板方法模式而不是策略模式？

**答案要点**（2 点）：
1. **需求不同**：需要定义算法骨架（执行流程固定），只是具体步骤不同
2. **继承关系**：BaseAgent → ReActAgent → ToolCallAgent 是"是什么"的关系，不是"能做什么"

**对比**：
- 模板方法：定义算法骨架，子类实现步骤（继承）
- 策略模式：定义算法接口，不同实现类（组合）

## 五、Spring 框架（重点）

### Q11：Spring Bean 的作用域有哪些？项目中如何使用的？

**答案要点**（4 个作用域）：
1. **singleton**（默认）：单例，整个容器一个实例 - 如 `LoveApp`、`YuManus`
2. **prototype**：每次获取创建新实例 - 如每次请求创建新的 `YuManus`
3. **request**：每个 HTTP 请求一个实例
4. **session**：每个 HTTP 会话一个实例

**项目使用**：
```java
@Component  // 默认 singleton
public class LoveApp { ... }

// 每次创建新实例
@GetMapping("/manus/chat")
public SseEmitter doChatWithManus(String message) {
    YuManus yuManus = new YuManus(allTools, dashscopeChatModel);  // 手动创建
    return yuManus.runStream(message);
}
```

### Q12：@Resource 和 @Autowired 的区别？

**答案要点**（3 点）：
1. **来源**：@Resource 是 JSR-250 标准，@Autowired 是 Spring 特有
2. **匹配方式**：@Resource 先按名称再按类型，@Autowired 先按类型再按名称
3. **项目使用**：统一使用 @Resource，更符合 Java 标准

**代码示例**：
```java
@Resource
private VectorStore loveAppVectorStore;  // 先按名称 loveAppVectorStore 匹配

@Autowired
private VectorStore loveAppVectorStore;  // 先按类型 VectorStore 匹配
```

### Q13：Spring AI 的 Advisor 机制是如何工作的？

**答案要点**（3 点）：
1. **作用**：类似 AOP 拦截器，在 AI 调用前后执行逻辑
2. **实现**：实现 `CallAroundAdvisor` 接口，重写 `aroundCall()` 方法
3. **执行顺序**：通过 `getOrder()` 控制执行顺序

**代码示例**：
```java
public class MyLoggerAdvisor implements CallAroundAdvisor {
    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request, CallAroundAdvisorChain chain) {
        log.info("请求：{}", request.userText());  // 前置处理
        AdvisedResponse response = chain.nextAroundCall(request);  // 调用下一个
        log.info("响应：{}", response.response().getResult().getOutput().getText());  // 后置处理
        return response;
    }
}
```

## 六、并发编程（高频）

### Q14：BaseAgent.runStream() 中为什么使用 CompletableFuture.runAsync()？

**答案要点**（3 点）：
1. **异步执行**：SSE 流式输出需要长时间运行，不能阻塞主线程
2. **非阻塞**：立即返回 SseEmitter，客户端可以接收流式数据
3. **异常处理**：通过 `sseEmitter.completeWithError()` 处理异常

**关键代码**：
```java
public SseEmitter runStream(String userPrompt) {
    SseEmitter sseEmitter = new SseEmitter(300000L);
    CompletableFuture.runAsync(() -> {  // 异步执行
        // 执行循环
        for (int i = 0; i < maxSteps; i++) {
            String result = step();
            sseEmitter.send(result);  // 流式发送
        }
        sseEmitter.complete();
    });
    return sseEmitter;  // 立即返回
}
```

### Q15：FileBasedChatMemory 是线程安全的吗？如何改进？

**答案要点**（3 点）：
1. **当前问题**：不是线程安全的，多线程并发访问可能丢失消息
2. **原因**：`getOrCreateConversation()` 和 `saveConversation()` 不是原子操作
3. **改进方案**：使用 `ConcurrentHashMap` + `synchronized` 或 `ReentrantReadWriteLock`

**改进代码**：
```java
private final Map<String, ReentrantReadWriteLock> locks = new ConcurrentHashMap<>();

public void add(String conversationId, List<Message> messages) {
    ReentrantReadWriteLock lock = locks.computeIfAbsent(conversationId, k -> new ReentrantReadWriteLock());
    lock.writeLock().lock();
    try {
        List<Message> messageList = getOrCreateConversation(conversationId);
        messageList.addAll(messages);
        saveConversation(conversationId, messageList);
    } finally {
        lock.writeLock().unlock();
    }
}
```

### Q16：为什么使用 static final Kryo 而不是每次创建？

**答案要点**（3 点）：
1. **性能**：Kryo 对象创建成本高，复用可提升性能
2. **线程安全**：Kryo 不是线程安全的，但通过同步块保证安全
3. **内存效率**：避免频繁创建对象，减少 GC 压力

**代码示例**：
```java
public static final Kryo kryo = new Kryo();  // 静态共享

static {
    kryo.setRegistrationRequired(false);
    kryo.setInstantiatorStrategy(new StdInstantiatorStrategy());
}

// 使用时同步
synchronized (kryo) {
    kryo.writeObject(output, messages);
}
```

### Q18：如何实现智能体的自动终止机制？

**答案要点**（3 点）：
1. **终止工具**：创建 `TerminateTool`，AI 判断任务完成时调用
2. **检测机制**：在 `act()` 中检查工具返回结果，如果调用了 `doTerminate`，设置 `state = FINISHED`
3. **循环退出**：`for` 循环条件 `state != FINISHED`，自动退出

**关键代码**：
```java
// TerminateTool
@Tool(description = "Terminate the interaction when the request is met")
public String doTerminate() {
    return "任务结束";
}

// ToolCallAgent.act()
boolean terminateToolCalled = toolResponseMessage.getResponses().stream()
        .anyMatch(response -> response.name().equals("doTerminate"));
if (terminateToolCalled) {
    setState(AgentState.FINISHED);  // 终止
}
```

## 八、性能优化（加分）

### Q20：如何优化 RAG 检索的性能？

**答案要点**（5 点）：
1. **文档切分**：使用 `TokenTextSplitter` 合理切分，避免文档过大

4. **批量处理**：批量向量化文档，减少 API 调用
5. **异步处理**：文档加载和向量化异步执行
